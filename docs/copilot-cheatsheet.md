# Copilot 指令速查卡

> 默认前缀：
> **按本仓库规范工作：先读取 AGENTS.md、.github/copilot-instructions.md 和相关 docs。**

适用场景：GitHub Copilot Chat / Agent、Cursor、Claude Code。

## 默认开发

```text
按本仓库规范工作：先读取 AGENTS.md、.github/copilot-instructions.md 和相关 docs，只做最小改动；先给计划，再实施，最后给验证步骤。
```

## 新功能开发

```text
按本仓库规范工作：先读取 AGENTS.md、相关 docs 和当前模块代码，给我一个最小实现计划；确认后直接实现，并补上必要测试与验证步骤。
```

精简版：

```text
按仓库规范实现这个需求，只做最小必要改动，不要顺手重构无关代码。
```

## 修 Bug

```text
按本仓库规范排查这个问题：先定位根因，再给最小修复方案，然后直接修改代码，并说明如何验证修复有效。
```

只诊断不改代码：

```text
按本仓库规范先诊断这个 bug，先不要改代码；给我根因、影响范围和最小修复方案。
```

## 先出方案

```text
按本仓库规范先不要写代码；给我这个需求的最小实施计划、涉及文件、风险点和验证方式。
```

多方案对比：

```text
按仓库规范先给我 2 到 3 个实现方案，按复杂度、可维护性和改动范围比较，最后推荐最小可行方案。
```

## 增量修改现有代码

```text
按本仓库规范基于现有实现做增量修改，尽量复用当前模块模式，不新开抽象，除非现有结构明显不适合。
```

严格沿用现有风格：

```text
按仓库规范修改这部分代码，严格沿用现有命名、目录和实现风格，不要引入新的架构模式。
```

## 写测试

```text
按本仓库规范为这次改动补必要测试，优先覆盖核心行为、边界条件和容易回归的分支。
```

先列测试方案：

```text
按仓库规范先给我这次改动应该补哪些测试用例，不要写实现。
```

只补最必要测试：

```text
按仓库规范只补最必要的测试，避免为了覆盖率堆砌低价值用例。
```

## Code Review

```text
按本仓库规范审查当前改动，重点看正确性、潜在回归、测试缺口、边界条件和是否违反架构约束。
```

严格版：

```text
按仓库规范做严格 code review：先列问题，再按严重程度排序，并给出最小修复建议。
```

如果仓库有 review prompt：

```text
使用 review.prompt.md 的标准审查当前 diff。
```

## 小范围重构

```text
按本仓库规范重构这段代码，目标是提升可读性和可维护性，不改变对外行为；先说明重构边界，再实施。
```

更克制版：

```text
按仓库规范做小范围重构，只整理当前函数或模块，不扩大到无关文件。
```

## 性能优化

```text
按本仓库规范分析这段代码的性能问题，先找瓶颈和证据，再给最小优化方案，避免过早优化。
```

直接优化：

```text
按仓库规范对这里做最小性能优化，保留现有行为，并说明优化前后应如何验证。
```

## 前端页面开发

```text
按本仓库规范实现这个页面，优先复用现有组件和样式模式，保持交互完整，并补充必要状态与异常处理。
```

表单页：

```text
按仓库规范实现这个表单页面，包含校验、提交态、错误态和成功反馈，只做最小必要交互。
```

## 后端接口开发

```text
按本仓库规范实现这个接口，包含请求校验、错误处理、返回结构和必要测试，尽量沿用现有路由与服务层模式。
```

只改接口相关部分：

```text
按仓库规范只修改这个接口相关的路由、服务和测试，不要扩散改动到无关模块。
```

## 类型 / Schema / 数据模型调整

```text
按本仓库规范调整这里的数据结构，保持上下游类型一致，并列出受影响的位置和需要同步修改的文件。
```

先查影响范围：

```text
按仓库规范检查这次类型或模型变更的影响范围，先列清单，再逐个修改。
```

## 文档更新

```text
按本仓库规范为这次改动补简洁文档，重点写用途、入口、关键约束和开发者最容易踩坑的点。
```

只更新文档：

```text
按仓库规范更新相关 docs，保持内容简洁，只补和当前改动强相关的信息。
```

## 提交前检查

```text
按本仓库规范对当前改动做提交前检查：列出我应该跑的命令、需要确认的风险点，以及这次改动是否还缺测试或文档。
```

顺手整理 PR 内容：

```text
按仓库规范总结这次改动，给我 PR 标题、变更摘要、风险说明和验证步骤。
```

## 两个高频控制句

先别改代码：

```text
先不要改代码；按本仓库规范分析问题，给我最小方案、涉及文件和验证方法。
```

直接执行：

```text
按本仓库规范直接实施最小可行方案，不要展开大段解释；完成后只给我改动摘要和验证步骤。
```

## 常用限制词

限制改动范围：

```text
只改与当前任务直接相关的文件。
```

限制风格漂移：

```text
沿用现有代码风格和目录结构，不引入新模式。
```

减少解释：

```text
少解释，直接做。
```

先计划再改：

```text
先给计划，等我确认再改。
```

直接实施：

```text
不用等我确认，直接按最小方案实施。
```

## 最简单的记忆法

只记四类：

- 做功能：实现这个需求
- 修问题：排查并最小修复
- 看方案：先不要写代码，先给计划
- 做检查：review / test / 提交前检查

然后统一在前面加一句：

```text
按本仓库规范工作
```
